<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| torch.load ← checkpoint file | Deserializing a pickled checkpoint executes arbitrary code if the file is untrusted |
| subprocess ← git | git rev-parse runs a local binary (no untrusted input) |
| (none network) | Offline library; no network, no auth, no multi-user surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering / Elevation | `torch.load` pickle in `load_checkpoint` | accept (low) | Resume checkpoint contains optimizer/RNG python objects -> must load with full pickle; restricted to the user's OWN trusted files. Documented as trusted-only in the module docstring. The slim inference checkpoint (Phase 8) will use `weights_only=True`. No untrusted checkpoint source exists in Phase 1. |
| T-01-05 | Denial of Service | non-Pascal / cu128+ torch on Kaggle | mitigate | `preflight_p100` runs a Pascal smoke op and raises a clear error naming the cu128+ kernel-drop risk before any long run starts. |
| T-01-06 | Tampering | resume re-seeds instead of restoring RNG state | mitigate | `load_checkpoint` restores generator STATE (set_rng_state/set_state/setstate); the kill-and-resume trajectory test (1e-6) gates against re-seed regressions. |
| T-01-07 | Information Disclosure | checkpoints / logs committed to git | accept (low) | `.gitignore` (from 01-01) covers `*.pt`, `checkpoints/`, `logs/`. No secrets serialized in Phase 1. |
</threat_model>
